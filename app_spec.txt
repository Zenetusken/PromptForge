<project_specification>
  <project_name>PromptForge v2 - Intelligent Prompt Optimization Engine</project_name>

  <overview>
    PromptForge is a self-adaptive prompt optimization webapp that transforms raw, unstructured prompts
    into properly structured, improved, and optimized versions. The system autonomously analyzes incoming
    prompts, detects their type and weaknesses, selects the optimal optimization strategy, applies it,
    and validates the result — all without human configuration.

    PromptForge runs exclusively on Anthropic's Claude models. On startup it auto-detects the best
    available provider: if the Claude Code CLI is installed and a Max subscription is present, it uses
    the claude-agent-sdk (zero API cost). If no subscription is detected but an ANTHROPIC_API_KEY
    environment variable is set, it falls back to the Anthropic Python SDK with direct API calls.
    There is no user-facing model selector — the system automatically routes each pipeline stage to the
    optimal Claude model based on task complexity (Haiku for fast codebase exploration, Sonnet for
    analysis and validation, Opus for deep strategy and rewriting).

    When a GitHub repository is linked, Stage 0 (Explore) runs as a fully agentic coding assistant
    before the main analysis pipeline. Instead of statically describing a prompt, the explore agent
    dynamically investigates the codebase using real tool calls — reading files, listing directories,
    searching for patterns — to build a CodebaseContext object. This context flows into all subsequent
    pipeline stages (Analyze through Validate), making optimizations genuinely codebase-aware.

    All pipeline stages stream in real-time to the frontend via HTTP Server-Sent Events (SSE).
    The MCP server uses the WebSocket transport protocol, enabling persistent bidirectional
    connections from Claude Code and any WebSocket-capable MCP client.

    The UI is a Developer Workbench — a VS Code-inspired 5-zone layout (Activity Bar, Navigator,
    Editor Groups, Inspector, Status Bar) with the industrial cyberpunk aesthetic. All pipeline
    stages stream in real time inside the Editor's Pipeline sub-tab, results are viewable as
    side-by-side or inline diffs, and every prompt has its own run history with per-stage traces.
  </overview>

      <llm_integration>
        Two mutually exclusive provider modes (auto-detected at startup, never user-selected):
          1. ClaudeCLIProvider — claude-agent-sdk, uses Max subscription via CLI (zero cost)
          2. AnthropicAPIProvider — anthropic Python SDK, requires ANTHROPIC_API_KEY

        Provider detection order:
          1. Check if `claude` CLI is available on PATH (shutil.which("claude"))
          2. Check ~/.claude/ for valid credentials/session (subscription present)
          3. If both true → use ClaudeCLIProvider
          4. Else if ANTHROPIC_API_KEY env var is set → use AnthropicAPIProvider
          5. Else → raise ProviderNotAvailableError with clear setup instructions

        Detection timeout:
          Each provider probe must complete within 5 seconds or be skipped (log at WARNING).
          Total auto-detection must not block startup for more than 10 seconds.

        Both providers implement the same LLMProvider ABC:
          complete(system, user, model) -> str
          stream(system, user, model) -> AsyncGenerator[str, None]
          complete_json(system, user, model, schema) -> dict
          complete_agentic(system, user, model, tools, max_turns, on_tool_call, output_schema) -> AgenticResult

        AgenticResult fields: text (str), tool_calls (list[dict]), output (dict | None),
          stop_reason ("end_turn" | "max_turns" | "tool_error" | "cancelled").
          Note: "tool_error" and "cancelled" are reserved contract values. Neither the
          Anthropic API nor the claude-agent-sdk currently surfaces these as stop reasons;
          both providers return "end_turn" or "max_turns" in practice.
        ToolDefinition fields: name, description, input_schema (JSON Schema dict),
          handler (async callable).

        The agentic method is the key addition for GitHub-grounded analysis.
        For ClaudeCLIProvider: uses claude-agent-sdk with in-process MCP server tools.
          No server-side schema enforcement — complete_json(schema=...) uses best-effort
          parse_json_robust() fallback. init.sh unsets CLAUDECODE before starting the
          backend; no API delegation occurs at runtime.
        For AnthropicAPIProvider: runs a multi-turn tool-calling loop via messages.stream()
          with adaptive thinking (Opus/Sonnet 4.6) and ephemeral prompt caching.
      </llm_integration>

      <model_orchestration>
        Internal model routing — never exposed as a user-facing picker.
        Each pipeline stage is assigned the optimal Claude model:

          Stage 0 (Codebase Explore): claude-haiku-4-5
            Rationale: File tree traversal and content reading is I/O-bound. Haiku is the
            fastest model and routes to the latest Haiku 4.5 snapshot via alias. Only runs
            if a GitHub repo is linked.
            Latency target: variable (up to 30 seconds, capped at 15 tool-call turns).

          Stage 1 (Analyze): claude-sonnet-4-6
            Rationale: Prompt classification benefits from stronger reasoning. Sonnet delivers
            better task_type and intent detection than Haiku at acceptable latency and cost.
            When repo is linked, Sonnet receives the Stage 0 CodebaseContext as input.
            Latency target: &lt;5 seconds.

          Stage 2 (Strategy): claude-opus-4-6
            Rationale: Deep reasoning for optimal framework selection.
            Latency target: &lt;10 seconds.

          Stage 3 (Optimize): claude-opus-4-6
            Rationale: Creative rewriting at maximum capability. Streams progressively.
            Latency target: &lt;20 seconds.

          Stage 4 (Validate): claude-sonnet-4-6
            Rationale: Quality assessment — needs reasoning but not full Opus depth.
            Latency target: &lt;5 seconds.

        Model aliases (single source of truth — aliases always route to the latest snapshot):
          MODEL_ROUTING = {
            "explore":   "claude-haiku-4-5",
            "analyze":   "claude-sonnet-4-6",
            "strategy":  "claude-opus-4-6",

      </model_orchestration>

    <provider_system>
      <llm_provider_interface>
        Both providers implement the same abstract base class in providers/base.py:

          class LLMProvider(ABC):

              @abstractmethod
              async def complete(self, system: str, user: str, model: str) -> str:
                  """Single-shot completion. Returns full response text."""

              @abstractmethod
              async def stream(self, system: str, user: str, model: str) -> AsyncGenerator[str, None]:
                  """Streaming completion. Yields text chunks as they arrive."""

              @abstractmethod
              async def complete_json(self, system: str, user: str, model: str, schema: type) -> dict:
                  """Structured JSON output. Uses 3-strategy fallback:
                     1. Parse raw response as JSON
                     2. Extract first ```json ... ``` code block, parse it
                     3. Extract first { ... } substring with regex, parse it
                  """

              @abstractmethod
              async def complete_agentic(
                  self,
                  system: str,
                  user: str,
                  model: str,
                  tools: list[ToolDefinition],
                  max_turns: int = 20,
                  on_tool_call: Callable[[str, dict], None] | None = None,
              ) -> AgenticResult:
                  """Agentic tool-calling loop. Runs until stop_reason is end_turn or
                     max_turns is reached. Calls on_tool_call(tool_name, tool_input)
                     as a side-effect hook for SSE streaming of tool activity.
                     Returns AgenticResult(text=final_text, tool_calls=[...]).
                  """

          @dataclass
          class ToolDefinition:
              name: str
              description: str
              input_schema: dict   # JSON Schema object
              handler: Callable[[dict], Awaitable[str]]  # async function(args) -> str

          @dataclass
          class AgenticResult:
              text: str            # final response after all tool calls
              tool_calls: list[dict]  # log of { name, input, output } dicts

          MODEL_ROUTING = {
              "explore":   "claude-haiku-4-5",
              "analyze":   "claude-sonnet-4-6",
              "strategy":  "claude-opus-4-6",
              "optimize":  "claude-opus-4-6",
              "validate":  "claude-sonnet-4-6",
          }
      </llm_provider_interface>

      <claude_cli_provider>
        File: providers/claude_cli.py

        Standard methods (complete, stream, complete_json):
          Use query() with ClaudeAgentOptions(system_prompt=system, model=model, max_turns=1).
          Collect all AssistantMessage TextBlocks from the async generator.

        Agentic method (complete_agentic):
          Uses create_sdk_mcp_server() to spin up an in-process MCP server with the provided tools.
          Each ToolDefinition becomes a @tool-decorated async function.
          Tool names follow: mcp__{server_name}__{tool_name}

          Key implementation notes:
          - prompt must be an async generator (not a plain string) when using custom MCP servers
          - Tool results are handled automatically by the SDK (no manual tool_result loop)
          - Use allowed_tools to restrict which tools Claude can call

          from claude_agent_sdk import query, ClaudeAgentOptions, tool, create_sdk_mcp_server, AssistantMessage, TextBlock

          async def complete_agentic(self, system, user, model, tools, max_turns=20, on_tool_call=None):
              # Build in-process MCP server from ToolDefinitions
              sdk_tools = []
              for td in tools:
                  # Must capture td in closure correctly
                  handler = td.handler
                  name = td.name
                  @tool(td.name, td.description, td.input_schema)
                  async def _tool_fn(args: dict, _handler=handler, _name=name, _on=on_tool_call) -> dict:
                      result = await _handler(args)
                      if _on:
                          _on(_name, args)
                      return {"content": [{"type": "text", "text": result}]}
                  sdk_tools.append(_tool_fn)

              mcp_server = create_sdk_mcp_server(name="pf-tools", version="1.0.0", tools=sdk_tools)
              allowed = [f"mcp__pf-tools__{td.name}" for td in tools]

              async def _prompt_stream():
                  yield {"type": "user", "message": {"role": "user", "content": user}}

              options = ClaudeAgentOptions(
                  system_prompt=system,
                  model=model,
                  max_turns=max_turns,
                  mcp_servers={"pf-tools": mcp_server},
                  allowed_tools=allowed,
              )
              full_text = ""
              tool_calls = []
              async for msg in query(prompt=_prompt_stream(), options=options):
                  if isinstance(msg, AssistantMessage):
                      for block in msg.content:
                          if isinstance(block, TextBlock):
                              full_text = block.text
              return AgenticResult(text=full_text, tool_calls=tool_calls)
      </claude_cli_provider>

      <anthropic_api_provider>
        File: providers/anthropic_api.py

        Standard methods (complete, stream, complete_json):
          Use client.messages.create() or client.messages.stream().
          For stream: collect from async with client.messages.stream() as s: async for chunk in s.text_stream.

        Agentic method (complete_agentic):
          Runs an explicit multi-turn tool-calling loop:
            1. Call client.messages.create(tools=[...], messages=[...])
            2. If stop_reason == "tool_use": execute all tool_use blocks, append tool_result
            3. Loop until stop_reason == "end_turn" or max_turns exceeded
            4. Return AgenticResult with final text

          Tool format for Anthropic API:
            {"name": td.name, "description": td.description, "input_schema": td.input_schema}

          Tool result message format (all results in one user message):
            {"role": "user", "content": [
              {"type": "tool_result", "tool_use_id": block.id, "content": result_str},
              ...
            ]}

          async def complete_agentic(self, system, user, model, tools, max_turns=20, on_tool_call=None):
              api_tools = [{"name": t.name, "description": t.description,
                            "input_schema": t.input_schema} for t in tools]
              tool_map = {t.name: t.handler for t in tools}

              messages = [{"role": "user", "content": user}]
              turns = 0
              while turns < max_turns:
                  turns += 1
                  response = await self._client.messages.create(
                      model=model, max_tokens=8192, system=system,
                      tools=api_tools, messages=messages,
                  )
                  messages.append({"role": "assistant", "content": response.content})

                  if response.stop_reason == "end_turn":
                      text = next((b.text for b in response.content if hasattr(b, "text")), "")
                      return AgenticResult(text=text, tool_calls=[])

                  if response.stop_reason == "tool_use":
                      results = []
                      for block in response.content:
                          if block.type == "tool_use":
                              if on_tool_call:
                                  on_tool_call(block.name, block.input)
                              result_str = await tool_map[block.name](block.input)
                              results.append({"type": "tool_result",
                                              "tool_use_id": block.id,
                                              "content": result_str})
                      messages.append({"role": "user", "content": results})

              return AgenticResult(text="", tool_calls=[])
      </anthropic_api_provider>
    </provider_system>

    <github_integration>
      <overview>
        GitHub integration is OPTIONAL. PromptForge works without it (standard 4-stage pipeline).
        When a repository is linked, Stage 0 (Codebase Explore) activates, making the Analyze
        stage aware of the actual codebase content.

        Two connection methods are supported:
          A. OAuth App — user clicks "Connect GitHub", goes through browser OAuth flow.
             Tokens stored encrypted in DB per user session.
          B. Personal Access Token (PAT) — user pastes a fine-grained PAT directly into the UI.
             Must have "Contents: Read" permission on target repos.
             Stored encrypted in DB.

        GitHub App is NOT used (too complex for a single-user or small-team tool).
      </overview>

      <token_storage>
        GitHub tokens are NEVER stored in plaintext.
        Encryption: cryptography.fernet.Fernet with GITHUB_TOKEN_ENCRYPTION_KEY env var.
        If GITHUB_TOKEN_ENCRYPTION_KEY is not set, one is auto-generated and written to
        data/.github_encryption_key (gitignored). On first startup only.

        Storage: SQLite github_tokens table (see database schema below).
        Retrieval: token is decrypted in-memory for each use, never logged.
      </token_storage>

      <oauth_flow>
        Enabled only when GITHUB_CLIENT_ID and GITHUB_CLIENT_SECRET are set in .env.
        If not set, only PAT method is available.

        OAuth App setup:
          - App name: PromptForge (or user-defined)
          - Homepage URL: http://localhost:8000 (or deployed URL)
          - Callback URL: http://localhost:8000/auth/github/callback
          - Scope: "repo" (private + public) or "public_repo" (public only, shown in UI)

        Flow:
          1. GET /auth/github/login
             - Generate CSRF state token, store in signed cookie (itsdangerous.TimestampSigner)
             - Redirect to: https://github.com/login/oauth/authorize?
                 client_id=...&scope=repo&state=<csrf_token>&allow_signup=false
          2. GitHub redirects to GET /auth/github/callback?code=CODE&state=STATE
             - Verify state matches cookie (anti-CSRF)
             - POST to https://github.com/login/oauth/access_token to exchange code
             - Fetch /user to get github_user_id and login
             - Encrypt and store token in github_tokens table
             - Set session cookie with github_user_id
             - Return 302 to / (frontend handles via session state)
          3. GET /auth/github/me — returns current auth status and connected user info
          4. DELETE /auth/github/logout — revokes and deletes the stored token

        Session management:
          FastAPI session via signed cookie (itsdangerous, SECRET_KEY env var).
          Session stores: {github_user_id, github_login, session_id}
          Each optimization request includes the session_id to look up the GitHub token.
      </oauth_flow>

      <pat_flow>
        POST /auth/github/pat
          Body: { "token": "github_pat_..." }
          - Validate token by calling GET https://api.github.com/user with it
          - If valid: encrypt and store, return user info { "login": "...", "id": ... }
          - If invalid: return 401 with error message
          - Token is associated with the current browser session

        DELETE /auth/github/pat — clears the stored PAT (same as logout)
      </pat_flow>

      <repository_linking>
        After GitHub auth, users can link a specific repository to a PromptForge session.
        The linked repo is used as codebase context for the next optimization run.

        GET /api/github/repos
          - Fetches repos the user has access to (via PyGithub g.get_user().get_repos())
          - Returns: [{ full_name, name, private, default_branch, description, language, size_kb }]
          - Cached for 5 minutes (avoid repeated API calls)

        POST /api/github/repos/link
          Body: { "full_name": "owner/repo", "branch": "main" }
          - Validates the repo is accessible with the stored token
          - Stores in linked_repos table (see schema)
          - Returns: { "linked_repo_id": uuid, "full_name", "branch" }

        GET /api/github/repos/linked — returns the currently linked repo (if any)
        DELETE /api/github/repos/unlink — removes the link (optimization still runs, no Stage 0)

        The frontend shows the linked repo in the Status Bar as a clickable badge: "⬡ owner/repo@main"
        Clicking it switches the Navigator to the GitHub activity view where the user can change
        or unlink the repo. There is no modal — all repo management is in the Navigator panel.
      </repository_linking>

      <repo_content_loading>
        When a repo is linked, Stage 0 needs access to repo content via tool calls.
        The tools fetch content on-demand from the GitHub API using the user's stored token.

        PyGithub usage pattern (sync, wrapped in anyio.to_thread.run_sync):
          from github import Github, Auth
          import anyio

          async def get_repo_tree(token: str, full_name: str, branch: str) -> list[dict]:
              def _sync():
                  g = Github(auth=Auth.Token(token))
                  repo = g.get_repo(full_name)
                  b = repo.get_branch(branch)
                  tree = repo.get_git_tree(b.commit.commit.tree.sha, recursive=True)
                  return [{"path": e.path, "sha": e.sha, "size": e.size}
                          for e in tree.tree if e.type == "blob"]
              return await anyio.to_thread.run_sync(_sync)

          async def get_file_content(token: str, full_name: str, sha: str) -> str:
              def _sync():
                  import base64
                  g = Github(auth=Auth.Token(token))
                  repo = g.get_repo(full_name)
                  blob = repo.get_git_blob(sha)
                  if blob.encoding == "base64":
                      return base64.b64decode(blob.content).decode("utf-8", errors="replace")
                  return blob.content
              return await anyio.to_thread.run_sync(_sync)

        File filtering — always exclude:
          Extensions: .png .jpg .jpeg .gif .ico .svg .woff .woff2 .ttf .eot .pdf
                      .zip .gz .tar .mp4 .mp3 .pyc .pyo .so .dll .exe .bin
          Directories: node_modules .git __pycache__ .venv venv dist build .next .nuxt vendor
          Max file size: 100KB per file (skip silently, note in tool output)
          Max files fetched per tool call: 20 (prevent runaway API consumption)

        Rate limit handling:
          Authenticated GitHub API: 5000 req/hr.
          PyGithub 2.x auto-retries on 403/429 with backoff.
          On rate limit hit during Stage 0, gracefully stop exploration and emit a warning SSE event.
      </repo_content_loading>

    <agentic_tool_calling>
      <overview>
        When a GitHub repository is linked, Stage 0 (Codebase Explore) runs before the
        standard 4-stage pipeline. This stage is agentic: the model uses real tool calls
        to explore the repository, read files, and search for patterns — exactly like a
        coding assistant with access to the codebase.

        The output of Stage 0 is a CodebaseContext object that is:
          - Passed to Stage 1 (Analyze) as additional input, grounding the analysis
          - Persisted in the optimization record (codebase_context_snapshot field)
          - Surfaced in the UI as an expandable "Codebase Context" panel

        The agentic explore stage is bounded:
          - Model: claude-haiku-4-5
          - Max tool-calling turns: 15
          - Max file reads per run: 20 files
          - Timeout: 30 seconds (if exceeded, continue pipeline with partial context)
      </overview>

      <tool_definitions>
        Five tools are available to the Stage 0 explore agent.
        These are defined in services/codebase_tools.py as ToolDefinition objects.

        TOOL: list_repo_files
          Description: "List all files in the linked repository. Returns the complete file tree
                        with paths, sizes, and SHA hashes. Call this first to understand
                        the repository structure before reading specific files."
          Input schema:
            { "type": "object", "properties": {
                "path_prefix": {"type": "string",
                                "description": "Filter to files under this path. Empty = all files."},
                "max_results": {"type": "integer", "default": 200,
                                "description": "Max entries to return. Default 200."}
            }, "required": [] }
          Handler: Calls get_repo_tree(token, full_name, branch), filters by path_prefix,
                   returns JSON list of { path, size_bytes } (omit SHA from output).

        TOOL: read_file
          Description: "Read the full content of a specific file from the linked repository.
                        Use the exact path from list_repo_files. For large files (>50KB)
                        only the first 200 lines are returned."
          Input schema:
            { "type": "object", "properties": {
                "path": {"type": "string", "description": "Exact file path as returned by list_repo_files"},
                "max_lines": {"type": "integer", "default": 200}
            }, "required": ["path"] }
          Handler: Looks up SHA from tree cache (or re-fetches tree), calls get_file_content(token, full_name, sha).
                   Returns file content truncated at max_lines lines.
                   Returns error message if file not found or is binary/too large.

        TOOL: search_code
          Description: "Search for a text pattern across all files in the linked repository.
                        Returns matching lines with file paths and line numbers.
                        Use for finding API calls, imports, function definitions, etc."
          Input schema:
            { "type": "object", "properties": {
                "pattern": {"type": "string", "description": "Literal string or simple pattern to search for"},
                "file_extension": {"type": "string",
                                   "description": "Filter to files with this extension (e.g. 'py', 'ts', 'md'). Empty = all."},
                "max_results": {"type": "integer", "default": 30}
            }, "required": ["pattern"] }
          Handler: Fetches all file blobs (up to 20), searches for pattern, returns matches.
                   Format: "file.py:42: matching line content"
                   Expensive — filters by extension first, exits early when max_results reached.

        TOOL: read_multiple_files
          Description: "Read up to 5 files at once. More efficient than calling read_file multiple times.
                        Returns each file separated by a clear header."
          Input schema:
            { "type": "object", "properties": {
                "paths": {"type": "array", "items": {"type": "string"},
                          "description": "List of file paths (max 5)"}
            }, "required": ["paths"] }
          Handler: Iterates paths, calls get_file_content for each, returns concatenated content.
                   Limits each file to 150 lines.

        TOOL: get_repo_summary
          Description: "Get a high-level summary of the repository: README content,
                        package/dependency files, and entry points. Always call this first
                        to get oriented before exploring specific files."
          Input schema: { "type": "object", "properties": {}, "required": [] }
          Handler: Automatically fetches and returns content of:
                   README.md (or README.rst), package.json (or pyproject.toml or Cargo.toml),
                   .env.example, CLAUDE.md (if present), and the root directory listing.
                   All returned in a single formatted block. Max 300 lines total.

        TOOL EXECUTION NOTE:
          All tool handlers are async functions with signature:
            async def handler(args: dict) -> str
          They use the GitHub token retrieved from DB for the current session.
          Errors are returned as strings (not raised) so Claude can reason about them.
      </tool_definitions>

      <stage_0_system_prompt>
        You are a codebase analysis assistant with access to a GitHub repository.
        Your goal is to build a rich understanding of this codebase that will help
        PromptForge optimize a user's prompt about it.

        The user's prompt is:
        ---
        {raw_prompt}
        ---

        Use your tools to explore the repository. Focus on:
        1. ALWAYS start with get_repo_summary to understand the project at a high level
        2. Read key source files relevant to what the user's prompt is asking about
        3. Identify: tech stack, key patterns, main abstractions, relevant APIs/functions
        4. Note anything in the codebase that the user's prompt is unclear or wrong about

        After exploring, return a JSON object:
        {
          "repo": "owner/repo",
          "tech_stack": ["Python", "FastAPI", "SQLAlchemy"],
          "key_files_read": ["path/to/file.py"],
          "relevant_code_snippets": [{"file": "...", "lines": "...", "context": "why relevant"}],
          "codebase_observations": ["The API uses REST not GraphQL", "Auth is JWT-based"],
          "prompt_grounding_notes": ["The prompt mentions X but the codebase uses Y instead"]
        }

        Be efficient. You have a maximum of 15 tool-calling turns. Prioritize breadth (repo summary +
        key files) over depth (reading every file). Stop as soon as you have enough context to
        meaningfully ground the user's prompt in the codebase.
      </stage_0_system_prompt>

      <sse_events_for_stage_0>

          { "type": "stage", "data": {"stage": "explore", "status": "started", "repo": "owner/repo"} }

      <codebase_context_object>
        After Stage 0 completes, a CodebaseContext object is constructed:

          @dataclass
          class CodebaseContext:
              repo: str                        # "owner/repo"
              branch: str                      # "main"
              tech_stack: list[str]            # ["Python", "FastAPI", ...]
              key_files_read: list[str]        # paths actually read by the agent
              relevant_snippets: list[dict]    # [{ file, lines, context }]
              observations: list[str]          # agent's codebase notes

    <!-- OPTIMIZATION PIPELINE (5 stages when repo linked, 4 without)       -->
    <!-- ================================================================== -->
    <optimization_pipeline>
      When no repo is linked: 4 stages (Analyze → Strategy → Optimize → Validate)
      When repo is linked:    5 stages (Explore → Analyze → Strategy → Optimize → Validate)

      <stage_0_explore>
        See agentic_tool_calling section above.
        Produces: CodebaseContext object passed to Stage 1.
        If Stage 0 times out or errors: pipeline continues without CodebaseContext.
      </stage_0_explore>

      <stage_1_analyze>
        MODEL: claude-sonnet-4-6
        PURPOSE: Classify the prompt and identify optimization opportunities.

        INPUT: Raw prompt + (optionally) CodebaseContext JSON from Stage 0
        OUTPUT (structured JSON):
          - task_type: "coding" | "analysis" | "reasoning" | "math" | "writing"
                     | "creative" | "extraction" | "classification" | "formatting"
                     | "medical" | "legal" | "education" | "general" | "other"
                     (14 types — matches chromatic_encoding table; frontend applies color directly)
          - weaknesses: list[string]
          - strengths: list[string]
          - complexity: "simple" | "moderate" | "complex"
          - recommended_frameworks: list[string]  (from the 10 available frameworks)
          - codebase_informed: boolean  (true if CodebaseContext was provided)

        SYSTEM PROMPT: Act as a prompt engineering expert. Evaluate the raw prompt against:
          1. Clarity — Is the intent unambiguous?
          2. Specificity — Are requirements concrete and measurable?
          3. Structure — Is there logical organization?
          4. Context — Is sufficient background provided?
          5. Constraints — Are boundaries and limitations defined?
          6. Output Format — Is the expected response format specified?
          7. Examples — Are illustrative examples included where helpful?
          8. Persona/Role — Is there a useful role assignment?
          If codebase context is provided: also evaluate the prompt's accuracy relative to the codebase.

        SSE event: { "type": "analysis", "data": { task_type, weaknesses, strengths, complexity, codebase_informed } }
      </stage_1_analyze>

      <stage_2_strategy>
        MODEL: claude-opus-4-6
        PURPOSE: Select and justify the optimal optimization framework combination.

        INPUT: Raw prompt + analysis result + (optionally) CodebaseContext
        OUTPUT (structured JSON):
          - primary_framework: string
          - secondary_frameworks: list[string] (0-2 additional)
          - rationale: string
          - approach_notes: string  (specific instructions for the optimizer)

        FRAMEWORKS (10 options):
          CO-STAR, RISEN, chain-of-thought, few-shot-scaffolding,
          role-task-format, structured-output, step-by-step,
          constraint-injection, context-enrichment, persona-assignment

        SSE event: { "type": "strategy", "data": { primary_framework, secondary_frameworks, rationale } }
      </stage_2_strategy>

      <stage_3_optimize>
        MODEL: claude-opus-4-6 (streaming)
        PURPOSE: Rewrite the prompt using the selected strategy.

        INPUT: Raw prompt + analysis + strategy + (optionally) CodebaseContext
        OUTPUT (structured JSON):
          - optimized_prompt: string
          - changes_made: list[string]
          - framework_applied: string
          - optimization_notes: string

        STREAMING: Streams token by token via SSE step_progress events.
        SSE events:
          { "type": "step_progress", "data": { "step": "optimize", "content": "chunk" } }
          { "type": "optimization", "data": { optimized_prompt, changes_made, framework_applied } }
      </stage_3_optimize>

      <stage_4_validate>
        MODEL: claude-sonnet-4-6
        PURPOSE: Score and verify the optimized prompt is genuinely better.

        INPUT: Original + optimized prompt + changes_made
        OUTPUT (structured JSON):
          - is_improvement: boolean
          - clarity_score: integer (1-10)
          - specificity_score: integer (1-10)
          - structure_score: integer (1-10)
          - faithfulness_score: integer (1-10)
          - conciseness_score: integer (1-10)
          - overall_score: integer (1-10) — ALWAYS computed server-side (weighted average)
          - verdict: string
          - issues: list[string]

        Score weights: clarity 20%, specificity 20%, structure 15%, faithfulness 25%, conciseness 20%
        overall_score = round(0.20*clarity + 0.20*specificity + 0.15*structure +
                              0.25*faithfulness + 0.20*conciseness)
        NEVER trust the LLM's arithmetic — compute the weighted average in Python, round to int.

        If is_improvement=false and overall_score &lt; 5: retry Stage 3 once with adjusted constraints.
        SSE event: { "type": "validation", "data": { scores, is_improvement, verdict } }
      </stage_4_validate>

      <sse_protocol>
        All SSE events follow this envelope on the wire:
          event: {event_type}\n
          data: {JSON string}\n\n

        Complete event sequence:
          "stage"        — { stage: "explore"|"analyze"|"strategy"|"optimize"|"validate", status: "started" }
          "tool_call"    — { tool, input, status } (Stage 0 only)
          "tool_result"  — { tool, input, preview, status } (Stage 0 only)
          "analysis"     — { task_type, weaknesses, strengths, complexity, codebase_informed }
          "strategy"     — { primary_framework, secondary_frameworks, rationale }
          "step_progress"— { step: "optimize", content: "chunk" }
          "optimization" — { optimized_prompt, changes_made, framework_applied }
          "validation"   — { is_improvement, scores: {...}, verdict, issues }
          "complete"     — { optimization_id, total_duration_ms }
          "error"        — { stage, error, recoverable }
          "rate_limit_warning" — { stage: "explore", message: "...", tools_called: N }
            NOTE: rate_limit_warning is non-fatal. Frontend displays a dismissible warning
            toast and continues streaming — it is NOT treated as an error. Emitted when the
            explore stage approaches its tool-call budget before completing.

        Frontend consumes via fetch() + ReadableStream (NOT EventSource — too limited):
          const res = await fetch('/api/optimize', { method: 'POST', ... });
          const reader = res.body.getReader();
          const decoder = new TextDecoder();
          let buffer = '';
          while (true) {
            const { value, done } = await reader.read();
            if (done) break;
            buffer += decoder.decode(value, { stream: true });
            const events = buffer.split('\n\n');
            buffer = events.pop();
            for (const raw of events) {
              const typeMatch = raw.match(/^event: (.+)$/m);
              const dataMatch = raw.match(/^data: (.+)$/m);
              if (typeMatch && dataMatch) {
                dispatch(typeMatch[1], JSON.parse(dataMatch[1]));
              }
            }
          }
      </sse_protocol>
    </optimization_pipeline>

        MCP session GitHub token:
          The MCP server maintains a per-session in-memory GitHub token set via github_set_token.
          This is separate from the web UI's OAuth flow — Claude Code users provide their PAT via tool call.
          Tokens are NOT persisted to DB from the MCP server (privacy: MCP is a local tool).

        MCP session lifecycle:
          Session scope: One WebSocket connection = one session. The token set is keyed by the
          WebSocket connection object and discarded on disconnect. No session IDs are exchanged —
          isolation is structural (per-connection state). The MCP server must NOT share token
          state across concurrent connections.

    <!-- ════════════════════════════════════════════════════════════════ -->
    <zero_effects_directive>
      The following are BANNED everywhere in the codebase, with zero exceptions:
        ✗ box-shadow with spread or blur: "0 4px 12px rgba(...)", "0 0 8px #00e5ff"
        ✗ text-shadow: any value
        ✗ filter: drop-shadow(): any value
        ✗ radial-gradient fading to transparent around elements
        ✗ animation: pulse / breathe / radiate / glow-pulse
        ✗ Class names containing "glow", variables named "--glow-*", comments saying "add glow"
        ✗ border: 2px solid (always 1px)
        ✗ color: white / #ffffff for body text (use var(--color-text-primary) = #e4e4f0)

      PERMITTED alternatives for depth and emphasis:
        ✓ box-shadow: inset 0 0 0 Npx [color] — for contour rings only
        ✓ border: 1px solid [color] — the ONE true accent technique
        ✓ background-color shift — opacity tier changes for elevation
        ✓ transform: translateY(-1px) — physical lift without visual bloom
        ✓ color / opacity increase on hover — text brightening, not glowing

      Vocabulary: "contour" (not glow), "tint" (not halo), "flash" (not pulse), "emission" (not radiance)
    </zero_effects_directive>

    <!-- 5-STATE INTERACTION MACHINE                                       -->

    <interaction_states>
      Every interactive element follows this 5-state lifecycle:

      Resting:  border-subtle, no accent color. Dark, receded, waiting.
      Hover:    contour intensifies one tier. Background tints. Text brightens. 200ms simultaneous.
      Active:   contour at max. Transform snaps to resting (translateY(0)). Border mutes.
      Focus:    1px solid rgba(0, 229, 255, 0.3) outline, offset 2px. ADDITIVE — overlays any state.
      Disabled: opacity 0.4, cursor not-allowed. No transitions. Snap instantly.

      State machine flow:
        Resting──hover──→Hover──press──→Active──release──→Hover──leave──→Resting
        Focus can overlay any state. Disabled bypasses all states.

      Contour intensity tiers:
        Micro:  1px border color shift — status dots, sidebar active tab
        Small:  1px color border + faint inset background — button hover, toggle
        Medium: 1px solid neon + opaque background (/12) — focus states, recommendation buttons
        Large:  inset 0 0 0 1px inner contour — card hover, prompt card focus
        Hero:   pure solid vector highlight + translateY(-1px) lift — primary button resting

      All multi-property transitions fire SIMULTANEOUSLY in a single declaration.
      Never stagger individual transition channels.
    </interaction_states>

    <anti_patterns>
      If any of these appear in code, it is a bug — fix immediately:
        ✗ box-shadow: 0 4px 12px rgba(...)     → use border: 1px solid + background shift
        ✗ text-shadow: 0 0 8px #00e5ff         → use brighter color or higher font-weight
        ✗ filter: drop-shadow(...)             → use border or outline
        ✗ animation: pulse 2s infinite         → use copy-flash (one-shot) or static contour
        ✗ radial-gradient(circle, #00e5ff, transparent) → use solid bg at low opacity
        ✗ Opacity decreasing on hover          → opacity always INCREASES on hover
        ✗ border: 2px solid                    → always 1px solid
        ✗ color: white / #ffffff in body       → use var(--color-text-primary) = #e4e4f0
        ✗ font-family: monospace for body      → mono ONLY for scores/badges/data
        ✗ Staggered transition declarations    → single transition: declaration, same duration
        ✗ .glow-effect / --glow-color in code  → rename to "contour", "tint", "flash"
        ✗ z-index: 999 or unlisted value       → use one of the 9 defined layers above
        ✗ Inter or JetBrains Mono for UI text  → use Geist / Geist Mono / Syne
        ✗ border-radius > 16px                 → max is 16px (large modals only)
        ✗ More than 2 gradient elements per viewport → use solid colors instead
