# Project Synthesis — Vision

> The system that builds software also builds a better version of itself.

---

## The Problem

Prompts are programs. They are the boundary layer between human intent and machine execution, and like any program, they range from barely functional to rigorously engineered. A casually written prompt and a precisely structured one can produce radically different outputs from the same model — the difference is rarely visible, always consequential.

But prompt engineering has no tooling infrastructure. No compiler catches logical ambiguity. No linter flags missing constraints. No version control surfaces why a prompt regressed between runs. Teams rediscover the same optimization patterns independently, accumulate the same failure modes silently, and have no institutional memory of what works at scale.

This problem is not just friction for individual engineers. It is a structural ceiling on AI-assisted development. Autonomous agents — systems that build software, run test suites, and make architectural decisions — are entirely driven by prompts written without systematic rigor. The quality of the prompt determines the quality of everything downstream.

Project Synthesis removes that ceiling.

## The Architecture

Three interlocking layers. Each is independently useful. Together, they form a closed loop that compounds in capability with every run.

### Layer 1 — Prompt Synthesis

The foundation. Every prompt enters a five-stage analysis pipeline:

| Stage | Function |
|---|---|
| **Explore** | Reads linked repository context via live tool calls — file trees, key files, patterns — building a CodebaseContext that flows into all subsequent stages |
| **Analyze** | Classifies prompt type, task domain, and complexity; identifies structural weaknesses and missing constraints |
| **Strategy** | Selects the optimal optimization framework from a library of proven techniques (Chain-of-Thought, Tree-of-Thought, COSTAR, RISEN, CRISPE, and others) |
| **Optimize** | Rewrites the prompt using the selected framework, informed by codebase context and analysis findings |
| **Validate** | Scores the result across multiple dimensions — clarity, specificity, coverage, grounding — and returns actionable improvement feedback |

Every run is logged. Every score is stored. Every diff is browsable. The developer workbench — a VS Code-inspired five-zone layout with real-time SSE streaming — makes the pipeline fully transparent rather than a black box.

This is the current production state of Project Synthesis.

### Layer 2 — Code Synthesis

The generative layer. Optimized prompts drive an autonomous multi-agent coding system. Agent sessions rotate through specialized roles — Initializer, Coder, Integrator, Validator — each session receiving a fresh context window and communicating via a structured scratchpad. The system builds full-stack applications from a specification, using browser-automated tests to verify correctness at each tier before advancing.

Layer 1 feeds Layer 2 directly: a well-optimized specification produces measurably fewer integration failures, higher first-pass test success rates, and more coherent architectural decisions. The coding agents are consuming the output of the optimization pipeline as their primary input.

This layer exists in the autonomous builder system and is being merged into Project Synthesis.

### Layer 3 — Self-Evolution

The adaptive layer. After each synthesis run — prompt optimization, code generation, or both — the system analyzes its own performance. What patterns produced the highest validation scores? Which optimization frameworks succeeded on which prompt types? Where did agent sessions stall? What code generation patterns correlated with test failures?

This knowledge is not discarded. It is encoded into the strategy selector, the framework library, the agent prompting templates, and the pipeline configuration. Each run improves the next, automatically, without human intervention.

The system versions its own improvements: patches for small optimizations (less than 10% impact), minor versions for significant gains (10–50%), major versions for fundamental changes that require human review. Historical runs validate each improvement before it is applied; a regression triggers automatic rollback.

## The Recursion

What makes Project Synthesis structurally different from a prompt optimization tool is that it is self-referential in a productive way.

The autonomous coding agents in Layer 2 are driven by prompts. Those prompts are optimizable by Layer 1. Layer 1's optimization strategies are themselves generated by prompts. Layer 3 learns from the outcomes and rewrites those strategies. The system that builds software is built from the same material it optimizes — and it gets to improve that material continuously.

This is not a metaphor. The practical consequence: every project built by the coding system generates training signal that improves the optimization pipeline for the next project. An improvement to the Explore stage propagates to every subsequent code generation run. A new optimization framework discovered through Layer 3 pattern analysis becomes available to every prompt in Layer 1 the next day.

The compound effect becomes significant quickly:

```
Run 1:   Baseline efficiency — strategy library reflects known patterns
Run 10:  First learned improvements applied — blockers prevented proactively
Run 50:  Execution time reduced 50–70%, blocker frequency 80–90%, first-pass success 90%+
Run 100: Strategies are codebase-specific and project-aware
```

These are engineering milestones, not research problems.

## Roadmap

### Phase 1 — Prompt Intelligence (Current)

Single-user prompt optimization with full pipeline visibility. GitHub codebase context. Real-time streaming. Per-dimension scoring. Diff view. Persistent history. MCP server with 13 tools exposing the full API to Claude Code and other MCP clients. Two LLM providers auto-detected at startup.

The workbench is production-ready. The pipeline is accurate. The foundation is solid.

### Phase 2 — Autonomous Construction

Full-stack application generation from optimized specifications. Multi-agent sessions with per-tier test validation, stagnation detection, and progress tracking. Human oversight at defined checkpoints — the system asks for review before major version increments or when it detects ambiguity in the specification.

The specification that enters Phase 2 has already been optimized by Phase 1. This is the compounding advantage: better inputs to the coding system, not just better tooling around it.

### Phase 3 — Continuous Self-Improvement

The evolution engine activates. Pattern detection runs asynchronously after each session. Improvements are staged, validated against historical runs (regression gate at 10%), and applied automatically for patches and minor versions. The strategy library grows. The framework selector becomes more accurate. Agent prompts tighten.

Version history is fully transparent. Every automatic improvement is logged with its evidence — the runs it was derived from, the metric delta it produced, the validation outcome that confirmed it.

### Phase 4 — Cross-Project Intelligence (Speculative)

Within a single installation, learnings aggregate across all projects on that instance. The strategy library becomes tuned to the organization's specific domains, codebases, and engineering patterns. A new project inherits the optimization knowledge of every previous project run on the same instance.

Cross-instance learning — sharing anonymized patterns across independent deployments — would require a separate infrastructure layer: opt-in telemetry, a federated pattern store, and mechanisms to prevent prompt or codebase data from leaking through the learning signal. This is a solvable engineering problem, not a research one, but it is out of scope until the single-instance evolution loop is mature. It is noted here as the natural next horizon, not a current commitment.

## The End State

A development platform where engineers express intent and the system handles rigor. Where the gap between a rough idea and a working, tested, deployed application is measured in hours, not weeks. Where the cost of that gap decreases with every run because the system is continuously learning how to close it faster.

Not a tool that makes prompts better. A platform that makes AI-assisted development systematically more reliable — and that improves itself each time it is used.

The name is not marketing. It is the architecture:

**Synthesis** — the combination of disparate elements into a coherent, higher-functioning whole. Prompt components synthesized into optimized instructions. Instructions synthesized into running software. Outcomes synthesized into improved strategies. Each cycle, more capable than the last.

## Open Source

Project Synthesis is free and open source software, licensed under the [Apache License 2.0](LICENSE). It is designed to be self-hosted: your prompts, your codebase context, and your optimization history stay on your infrastructure. Nothing is sent to a central service.

Being open source here means something specific. The strategy library, the optimization frameworks, the pipeline architecture, and the evolution engine are all inspectable, forkable, and contributable. Improvements to these components — new optimization frameworks, better evaluation rubrics, more accurate strategy selectors — ship back to every installation through the normal open source update cycle. The community improves the codebase; each deployment benefits from those improvements on its own terms, on its own data.

This is the right model for a tool that reads your code and your prompts. Transparency and local control are not compromises. They are the point.
